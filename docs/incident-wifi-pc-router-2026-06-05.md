# Инцидент: нестабильный Wi‑Fi на ПК и недоступность Duangdeehouse2

**Дата:** 2026-06-04 … 2026-06-05  
**Оборудование:** ПК (Killer Wireless-n/a/ac 1535), роутер Duangdeehouse2, XIAO ESP32-S3  
**Статус на момент отчёта:** ПК и роутер восстановлены; плата XIAO требует отдельной проверки (IP / питание)

---

## Краткое резюме

После отладки Wi‑Fi прошивки XIAO пользователь столкнулся с:

- периодическим «отключением» беспроводного адаптера на ПК (мастер устранения неполадок: *«Адаптер "Беспроводная сеть" отключен»*);
- невозможностью подключиться к **Duangdeehouse2** с ПК и других устройств;
- ощущением, что ПК «перегружает роутер запросами».

**Вывод:** проблема была **на стороне ПК и нагрузки на роутер во время отладки**, а не в «поломке сетевой карты» командами отключения адаптера. Профили Wi‑Fi домашней сети **пропали из Windows**, ПК ушёл на чужую сеть; службы **Killer** усиливали обрывы. Параллельно ESP32 и скрипты диагностики создавали **высокую нагрузку на роутер** (повторные handshake, сканы, массовый ping).

## Хронология (логика, не поминутно)

1. **Прошивка build 11** — I2C ToF на GPIO20 → пропал USB CDC (COM); Wi‑Fi при этом мог работать.
2. **Исправление build 12+** — ToF перенесён на GPIO8; восстановлен COM; отладка Wi‑Fi, OTA, Serial.
3. **Build 13 (временно)** — при неудаче STA поднимался AP `xiao-robot` (откатили в build 14).
4. **Много переподключений ESP32** — в Serial: `reason=14` (4-way handshake timeout), `reason=36`; роутер мог временно блокировать MAC платы.
5. **Диагностика с ПК** — многократные `netsh wlan show networks`, ping, ARP-скан; открытие COM → **аппаратный reset** платы → снова цикл Wi‑Fi.
6. **Профиль Duangdeehouse2 исчез** из Windows → ПК не мог вернуться на домашнюю сеть.
7. **2026-06-05** — скрипт восстановления `tools/fix_pc_wifi_dihouse.ps1`: профили восстановлены, Killer-оптимизация отключена, ПК на `Duangdeehouse2  2.4GHz`, роутер `192.168.9.1` отвечает на ping.

## Причины (разбор)

### 1. Пропажа профиля Duangdeehouse2 в Windows

В сохранённых сетях остались только посторонние SSID (в т.ч. `Duangrudee House1-5G`, `DD_House`, `GATE_5GHz`). Без профиля Windows **не подключается** к дихаусу, даже если точка в эфире есть.

*Прямой команды «удалить профиль Duangdeehouse2» в задокументированных шагах не было; причина пропажи не установлена однозначно (возможны сброс WLAN-профиля, стороннее ПО Killer, ручные действия в системе).*

### 2. Killer Wireless + роуминг

На ПК:

- **Killer Wifi Optimization Service**
- **KNDBWM** (Dynamic Bandwidth Management)
- **Killer Analytics Service**

При агрессивном роуминге и энергосбережении адаптер может **кратковременно отключаться** на уровне драйвера — совпадает со скриншотом troubleshooter.

**Не отключалось:** сам адаптер через `Disable-NetAdapter` / `Disable-PnpDevice` для Killer (только USB Espressif `VID_303A` в `tools/xiao_recover_and_flash.ps1`).

### 3. Нагрузка на роутер во время отладки

| Источник | Эффект |
|----------|--------|
| ESP32: цикл `WiFi.begin` при `reason=14` | Много попыток WPA handshake → нагрузка на CPU роутера / anti-bruteforce |
| `netsh wlan show networks` (часто) | Активное сканирование на Killer — обрывы текущего соединения |
| `tools/xiao_find_lan.py` (до правки) | **40** параллельных `ping` по `192.168.9.2–59` |
| Открытие COM (native USB CDC) | Reset платы → новый DHCP / новый handshake |

Совокупность могла **подвесить или перезагрузить** слабый роутер — отсюда «ни одно устройство не цепляется».

### 4. Плата XIAO — отдельно от «поломки ПК»

- Прошивка **build 14 / 1.2.3** — режим **только STA** (без AP fallback build 13).
- Сеть и пароль: `xiao_cam_stream/secrets.h` (`XIAO_WIFI_SSID_1/2`, `kWifiPass`).
- Плата **не может** ронять Wi‑Fi на ПК; она клиент роутера, как телефон.
- После восстановления ПК: роутер OK; **плата на `192.168.9.11` не отвечала** (возможны другой IP, выключена, не в Wi‑Fi).

### 5. Отдельная аппаратная ошибка (ранее): GPIO19/20 и USB

Использование **GPIO19/20** под I2C ToF отключает линии **USB D-/D+** на ESP32-S3 → пропадает COM. Исправлено в `drive_config.h` (`XIAO_TOF_SDA = 8`), проверка в `xiao_tof.h` (`#error` при GPIO19/20).

## Что сделано для исправления

### Скрипт `tools/fix_pc_wifi_dihouse.ps1` (от администратора)

1. Остановить и отключить автозапуск: Killer Wifi Optimization, KNDBWM, Killer Analytics.
2. Отключить энергосбережение Wi‑Fi-адаптера; роуминг → **Lowest** (если свойство доступно).
3. `connectionmode=manual` для `Duangrudee House1-5G` и `xiao-robot`.
4. Восстановить XML-профили:
   - `Duangdeehouse2  2.4GHz` / `Duangdeehouse2 2.4GHz`
   - `Duangdeehouse2  5GHz` / `Duangdeehouse2 5GHz`
5. Одна попытка `netsh wlan connect` на 2.4 GHz (без циклического сканирования).

### Правка `tools/xiao_find_lan.py`

- Параллельные ping: **40 → 8** потоков (комментарий в коде: защита роутера).

### Прошивка

- Откат AP fallback: **build 14 / 1.2.3** (чистый STA к Duangdeehouse2).

### Проверка после фикса (2026-06-05)

| Проверка | Результат |
|----------|-----------|
| SSID на ПК | `Duangdeehouse2  2.4GHz` |
| Шлюз `192.168.9.1` | ping 3/3, 0% loss |
| Killer Optimization / KNDBWM / Analytics | Stopped, Disabled |
| Killer Network Service | Running (базовый стек, норма) |
| `192.168.9.11` (плата) | Нет ответа (нужна проверка питания / IP) |

## Текущие рекомендации

### ПК

- Не запускать часто `netsh wlan show networks` / полный ARP-ping подсети.
- При повторении обрывов: снова `tools/fix_pc_wifi_dihouse.ps1` **от имени администратора**.
- Держать автоподключение к **Duangdeehouse2**; чужие SSID — manual.
- Обновить драйвер Killer с сайта Intel/Killer при стабильных обрывах.

### Роутер Duangdeehouse2

- При «мертвом» Wi‑Fi на всех клиентах: **питание роутера 30 с**, затем включить.
- Зафиксировать канал 2.4 GHz (1 / 6 / 11), не «авто», если много соседей.
- При блокировке MAC платы: перезагрузка роутера или снятие блокировки в веб-интерфейсе.

### XIAO ESP32-S3

- ToF: **SDA=GPIO8, SCL=GPIO9**; не использовать GPIO19/20 под I2C.
- Питание: не одновременно USB + батарея на общую землю без схемы OR-ing (риск сгорания платы — см. обсуждение в чате).
- Телеметрия: `py -3 tools/xiao_serial_telemetry.py --port COM6 --http 8897` — не открывать COM во время проверки Wi‑Fi по LAN (reset по DTR).
- Найти IP: один раз `py -3 tools/xiao_find_lan.py` (≤8 ping), не в цикле.
