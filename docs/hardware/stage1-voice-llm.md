# Этап 1: говорящий робот на облачном LLM — детальный план

> Конкретный собираемый план: **поговорить с роботом по-русски, он понимает и едет по команде**,
> на текущем железе (существующая прошивка с `/drive` и `/telemetry`, динамик MAX98357A), без Pi.
> Мозг — облачный realtime-LLM. Зрения и памяти тут ещё нет (это Этап 2). См. общую картину в
> [llm-brain-2026.md](llm-brain-2026.md).

## Цель и рамки Этапа 1

- ✅ Робот **слышит голос** (микрофон) и **отвечает голосом** (динамик).
- ✅ LLM **вызывает функции робота** (ехать/повернуть/стоп/статус) → робот двигается.
- ✅ Русский — нативно (облачная realtime-модель).
- ✅ **Прошивку робота не меняем** — переиспользуем `/drive`, `/telemetry`.
- ❌ Без зрения, без памяти, без автономии (отдельные этапы).

## Архитектура — 3 узла

```
   ГОЛОС (ESP32-S3 #2)            МОЗГ-РЕЛЕЙ (Python на LAN)         ТЕЛО (ESP32 робота)
   ┌───────────────────┐  WS/PCM  ┌──────────────────────────┐ HTTP ┌──────────────────┐
   │ I2S-микрофон  ─────┼─────────►│ • держит API-ключ        │      │ существующая     │
   │ MAX98357A+динамик ◄┼──────────┤ • мост аудио ESP↔облако  │─────►│ прошивка:        │
   │ Wi-Fi, Opus/PCM    │          │ • tools робота (drive…)  │      │ /drive /telemetry│
   └───────────────────┘          │ • безопасность (клампинг)│      │ watchdog 450 мс  │
                                   └───────────┬──────────────┘      └──────────────────┘
                                               │ WSS
                                    ┌──────────▼───────────┐
                                    │ OpenAI gpt-realtime / │  слышит→думает→говорит,
                                    │ Gemini Live (S2S)     │  эмитит function_call'ы
                                    └───────────────────────┘
```

**Почему 3 узла, а не один:** на XIAO робота все пины заняты приводом, и тащить туда ещё realtime-
аудио+WebRTC нереально. Голос выносим на **отдельный ESP32-S3** (у него свободные пины под I2S), а
тело-ESP32 остаётся как есть. **Релей** (маленькая программа на любом устройстве в той же Wi-Fi)
держит ключ, мостит аудио и исполняет команды — он в LAN, поэтому достаёт `/drive` робота.

## Узел 1 — голосовой ESP32-S3 (железо + прошивка)

**Рекомендуемое железо:** **reSpeaker Lite Voice Kit (~$30)** — в нём уже **XIAO ESP32-S3 + 2-мик
массив + XMOS XU316 (аппаратный AEC)**. AEC важен: робот говорит и не слышит сам себя (без эха).
Выход звука — на твой **MAX98357A** + динамик 4–8 Ω.
- Альтернатива дешевле: любой ESP32-S3 + I2S-микрофон **ICS-43434 (~$6)** + MAX98357A. Без AEC —
  возможно самовозбуждение/эхо, лечится программно/разнесением мик-динамик.

**Прошивка голосового узла** (за основу — **ElatoAI** или **xiaozhi-esp32**):
- I2S-вход: микрофон → буфер PCM16 16 кГц → отправка в релей по WebSocket (или Opus для экономии трафика).
- I2S-выход: приём аудио от релея → MAX98357A → динамик.
- Wi-Fi STA (та же сеть, что робот и релей); опц. кнопка push-to-talk или VAD/wake-word.
- Пины (на голосовом ESP32, не на роботе): I2S-мик (BCLK/WS/DIN ×3) + I2S-amp (BCLK/LRC/DOUT ×3) —
  на отдельной плате пинов хватает.

## Узел 2 — мозг-релей (Python на LAN) — ядро системы

Это главное, что ты пишешь. Любой всегда-включённый комп в Wi-Fi (ноутбук сейчас, Pi/мини-ПК позже).
Функции:
1. **WebSocket-сервер** для голосового ESP32 (приём mic-аудио, отправка ответного аудио).
2. **Клиент realtime-API** (OpenAI gpt-realtime по WSS, или Gemini Live): передаёт mic-аудио, получает
   ответное аудио + события `function_call`.
3. **Конфиг сессии:** `system prompt` = **персона робота** (на русском) + правила безопасности; список **tools**.
4. **Исполнение tools:** на `function_call` → HTTP-запрос к роботу (`GET /drive?...`, `/telemetry`) →
   возврат результата модели.
5. **Слой безопасности** (см. ниже): клампинг скоростей, таймеры движений, проверка bumper/тока.

Почему релей, а не ключ на ESP32: безопасно (ключ не на устройстве), ESP32 разгружен (только аудио),
и релей в LAN достаёт робота. *(Вариант «без компа»: ключ на голосовом ESP32, он сам коннектится к
облаку и сам дёргает `/drive` робота — работает, но тяжелее и менее безопасно.)*

## Узел 3 — тело (без изменений)

Существующая прошивка уже даёт всё нужное:
- `GET /drive?l=-255..255&r=...` — привод, `GET /drive?stop=1` — стоп.
- `GET /telemetry` — JSON: `enc_l/enc_r` (энкодеры!), `bumper`, `us_cm`, `drive_*`, заряд (если добавишь INA).
- **Watchdog 450 мс** — если команды перестали идти, моторы сами стопаются (наша страховка).

## Tools — «руки» LLM (схемы + маппинг)

Определяешь в realtime-сессии. Реализация — в релее (HTTP к роботу). Дифф-привод: `l=v−w, r=v+w`.

| Tool (что модель вызывает) | Параметры | Реализация в релее |
|---|---|---|
| `move(direction, distance_cm)` | forward/back, 0–200 | по энкодерам: слать `/drive` пока Δenc не достигнет цели, затем `stop` |
| `turn(degrees)` | −180…180 | по разнице энкодеров бортов (или по времени), затем `stop` |
| `drive(left, right, ms)` | −150…150, ≤2000 | слать `/drive?l=&r=` каждые ~150 мс в течение `ms`, затем `stop` |
| `stop()` | — | `/drive?stop=1` |
| `get_status()` | — | `/telemetry` → вернуть заряд/энкодеры/bumper/препятствие |

⚠️ Из-за watchdog 450 мс «ехать» = **релей повторяет** команду в цикле на время действия, потом стоп.
Это и есть безопасность: LLM просит **ограниченное** действие, релей исполняет таймером, watchdog добивает.

## Безопасность (обязательно даже на Этапе 1)

- **Клампинг скоростей** в релее: например, максимум ±150 из ±255 (LLM не разгонит на полную).
- **Лимит длительности** одного движения (`ms ≤ 2000`); долгое движение = серия коротких с проверками.
- **Проверка препятствий:** перед/во время `move` читать `/telemetry` — `bumper`/`us_cm` → стоп.
- **Allowlist tools**, валидация параметров (тип/диапазон) до отправки.
- **`stop()` всегда доступен**; физический выключатель питания моторов.
- LLM **не управляет ШИМ напрямую** — только через эти безопасные tools.

## Голос: облако, русский, персона

- **OpenAI gpt-realtime** (WebRTC/WSS): function calling + image input (на будущее), ~200–300 мс,
  ~$0.05–0.46/мин. Или **Gemini Live** — аудио-вход ~10× дешевле + нативное видео (пригодится в Этапе 2).
- **Русский** — из коробки. **Персона** задаётся system-prompt'ом, например:
  > «Ты — Деньгожуй, дружелюбный колёсный робот. Отвечай коротко и по-русски. Можешь ехать, поворачивать
  > и сообщать статус через инструменты. Перед движением убедись, что путь свободен (get_status). Не
  > превышай безопасную скорость. Если просят странное/опасное — откажись и объясни.»

## Латентность и стоимость

- Диалог speech-to-speech: **~200–300 мс** до ответа.
- Стоимость: ~**$0.05–0.10/мин** с prompt-caching (OpenAI), без кэша до $0.46/мин; Gemini дешевле по аудио.
  Для эпизодического общения — копейки. Интернет обязателен.

## Порядок сборки (по вехам)

1. **M0 — мозг без голоса (быстрый proof):** релей на ноутбуке, говоришь в микрофон **ноутбука** →
   tools дёргают `/drive` робота → робот едет. Подтверждает связку LLM↔tools↔робот **за вечер**, без
   паяния. (Аудио — через realtime-SDK на ПК.)
2. **M1 — голос на роботе:** поднять голосовой ESP32 (reSpeaker Lite): тест I2S-мик + MAX98357A
   (loopback — слышишь себя) → подключить к релею → говоришь **через робота**.
3. **M2 — tools + безопасность:** реализовать `move/turn/stop/get_status` с энкодерами, клампинг,
   проверку bumper/препятствий, таймеры.
4. **M3 — персона + полиш:** русский system-prompt, push-to-talk или wake-word, аккуратные ответы,
   обработка отказов.

## Открытые решения (что выбрать перед стартом)

1. **Облако:** OpenAI gpt-realtime (лучший tool-calling, дороже) vs Gemini Live (дешевле аудио, видео).
2. **Мозг:** релей на ПК/ноуте в LAN (рекомендую) vs ключ прямо на голосовом ESP32 (без компа).
3. **Голосовое железо:** reSpeaker Lite (с AEC, рекомендую) vs дешёвый I2S-мик ICS-43434 + MAX98357A.
