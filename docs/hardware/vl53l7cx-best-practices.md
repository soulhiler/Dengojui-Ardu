## 1. Конфигурация сенсора

| Параметр | Факт | Источник |
|----------|------|----------|
| Разрешение/частота | **4×4 → до 60 Гц, 8×8 → жёстко до 15 Гц** (ULD отвергает 8×8 > 15 Гц). «60 Гц» в маркетинге ST — это про 4×4. | ULD ref; ST product page |
| Порядок | Сначала `set_resolution`, потом `set_ranging_frequency_hz` (предел зависит от разрешения) | UM3038 §4.x |
| Режимы | **AUTONOMOUS — дефолт.** CONTINUOUS: VCSEL включён весь период → лучшая дальность и помехозащита, «advised for fast ranging / high performance», больше энергии. AUTONOMOUS: VCSEL только на окно integration time, экономия растёт при низкой частоте. | UM3038 §4.5; макросы `CONTINUOUS=1, AUTONOMOUS=3` из исходника |
| Integration time | **Только в AUTONOMOUS.** Дефолт **5 мс**, диапазон ~2–1000 мс ⚠. В 8×8 — **4 интеграции**; сумма всех + 1 мс должна быть < периода кадра, иначе период авто-растягивается. | UM3038 §4.5/4.6 |
| Sharpener | Дефолт **5%** (драйвер) / **14%** (поздняя ревизия UM3038) ⚠, диапазон 0–99%. Срезает «растекание» сигнала сильной цели в соседние зоны. Выше % → резче разделение близких целей, но рискует стереть слабые дальние. | ULD ref; ST community |
| Target order | **Дефолт STRONGEST** (`CLOSEST=1, STRONGEST=2`). **Для избегания препятствий лучше CLOSEST** — ближайшая цель в зоне важнее самой яркой. | исходник `vl53l7cx_api.h` |
| Targets/zone | `VL53L7CX_NB_TARGET_PER_ZONE` дефолт **1**, билд-тайм макрос, 1–4. Рост → больше RAM и байт по I2C. | исходник `vl53l7cx_platform_config_default.h` |
| FoV | **60°×60° (квадрат, по оси) ≈ 90° по диагонали** — широкоугольный вариант (шире L5CX/L8CX). В 8×8 это **~7.5°/зону по оси**. | ST datasheet; community |
| Дальность | ~**350 см** (white 88% / grey 17% по Munsell). | datasheet; Pololu |
| Кросстолк | Гистограммный алгоритм даёт иммунитет к кросстолку через стекло **выше 60 см**; **ниже 60 см** кросстолк может превысить реальный сигнал → ложные «слишком близкие» цели. Нужна Xtalk-калибровка при наличии защитного стекла (цель ≥600 мм). | ST product page; UM3038 |

**Профили (рекомендация ST + сообщества):**
- **Манёвр/скорость:** 4×4, CONTINUOUS, 30–60 Гц, target order CLOSEST.
- **Точное обнаружение препятствий:** 8×8, ~15 Гц, CONTINUOUS, CLOSEST, sharpener выше если препятствия рядом со стеной.
- **Макс. дальность:** CONTINUOUS, 5–10 Гц (накопление сигнала), 8×8 ок, sharpener умеренный (не срезать слабые дальние). Не опускаться ниже ~5 Гц при целях <5 см.
- **Энергосбережение:** AUTONOMOUS, низкая частота, минимальная integration time.

## 2. I2C и интеграция на ESP32

- **Скорость шины:** сенсор поддерживает до **1 МГц (Fast-mode Plus)**. 400 кГц — безопасный дефолт; **1 МГц рекомендуют, чтобы срезать долгий init** (загрузка прошивки). ESP-IDF-порт работает на 1 МГц. Подвох ESP32: `Wire.setClock(1000000)` не всегда соблюдается ядром — проверять реальную частоту SCL.
- **Init / загрузка прошивки:** у сенсора **нет своей NVM** — хост каждый power-on заливает **~84 КБ прошивки по I2C** в волатильную RAM сенсора (на стороне MCU блоб лежит в `vl53l7cx_buffers.h`, ~84 КБ flash). `begin()`/init **до ~10 с на 400 кГц**, на 1 МГц ~вдвое быстрее. Наши ~2–3 с — это поведение 400 кГц. Частые `init fails` обычно = провода/подтяжки/буфер, а не чип.
- **Опрос vs прерывание:** `check_data_ready()` в цикле (просто) vs пин **GP1/INT** как GPIO-прерывание (ниже джиттер, не жжёт I2C). Прерывание GP1 **самосбрасывается через ~100 мкс** — в ISR ставить флаг, читать в loop. При наших 15–60 Гц (бюджет 16–67 мс) хватает обоих.
- **LPn:** гейт I2C-связи. **Должен быть HIGH**, чтобы сенсор отвечал; LOW — отключает I2C этого сенсора. На брейкаутах подтянут к VDD. Несколько сенсоров на шине: держать LPn LOW у всех, кроме программируемого, и менять адрес через `set_i2c_address()` (дефолт 0x52 8-бит / 0x29 7-бит).
- **Надёжность I2C:** известная беда — **SDA залипает в 0** (рассинхрон из-за большого объёма данных). Восстановление: дёрнуть LPn/reset ~10 мс или выдать 9 тактов SCL. **ESP32 имеет таймаут на clock-stretching ~10–13 мс** — поднять I2C timeout, иначе зависание/WDT. Внешние подтяжки, короткие провода. ⚠ У нас **LPn на перемычке к 3.3V** — программно восстановить залипшую шину не сможем.
- **RAM/буферы:** `VL53L7CX_Configuration` ~2.3 КБ (temp_buffer 1024 + offset 488 + xtalk 776), `VL53L7CX_ResultsData` ~1.4 КБ (массивы `[64]`). На ESP32 советуют стек задачи **≥7 КБ**. Arduino-Wire на ESP32 = 128 байт (ок; ULD всё равно чанкует). Библиотека `stm32duino/VL53L7CX` официально заявляет `architectures=stm32, sam` — **ESP32 не тестируемая платформа** (компилируется и работает, но «комьюнити-практика»).

## 3. Интерпретация данных

**Таблица `target_status`** (общая для L5CX/L7CX):

| Код | Значение |
|-----|----------|
| 0 | данные не обновлены |
| 1 | сигнал слишком слаб (массив SPAD) |
| 2 | target phase |
| 3 | sigma-оценка слишком высока |
| 4 | consistency failed |
| **5** | **дальность валидна (100% уверенности)** |
| 6 | wrap-around не выполнен (часто на первом замере; ~50%) |
| 7 | rate consistency failed |
| 8 | сигнал слишком слаб для текущей цели |
| **9** | **валидна с большим импульсом (возможно слитая цель; ~50% по UM3038)** |
| 10 | валидна, но цели на прошлом замере не было |
| 11 | measurement consistency failed |
| 12 | цель размыта другой из-за sharpener |
| 13 | цель есть, но данные несогласованы (часто у вторичных целей) |
| 255 | цели нет (если включён nb_target_detected) |

- ⚠ **Расхождение ST:** исходник драйвера говорит «5 и 9 = OK», UM3038 даёт 9 лишь ~50% и добавляет 6 как ~50%. Строгий фильтр: брать **5** (и опц. 9); 6 — низкая уверенность.
- **Поля для фильтрации шума:** `signal_per_spad` (kcps/spad — низкий = ненадёжно), `ambient_per_spad` (фон/засветка — высокий = солнце/IR-помеха), `range_sigma_mm` (СКО дистанции — высокий = шумно), `reflectance` (%). Рекомендуемый фильтр: `nb_target_detected>0 AND status∈{5,9} AND signal достаточен AND sigma мал`.
- **Индексация мультицели:** `distance_mm[zone * NB_TARGET_PER_ZONE + target]`. `nb_target_detected[zone]` и `ambient_per_spad[zone]` — только по зонам. Минимальное различимое расстояние между двумя целями ≈ **600 мм**.
- **Температура:** в струкуре есть `silicon_temp_degc`, **прошивка сама компенсирует** тепловой дрейф.

## 4. Алгоритмы для робота поверх 8×8

- **Препятствия:** маска по `status∈{5,9}` → флаг зоны ниже порога стопа (~15–30 см у хобби-роботов). Свернуть 8 столбцов в горизонтальные секторы (≈7.5°/столбец), взять мин/среднее по столбцу, рулить к столбцам с наибольшей чистой дистанцией — это по сути **Vector Field Histogram** (полярная гистограмма плотности, едем в самую широкую «долину»). У ST есть пример `Thresholds_Detection` (аппаратное прерывание при пересечении порога зоной).
- **Отсев пола / обрыв (cliff):** наклонить сенсор вниз, для каждой строки заранее посчитать ожидаемую дистанцию до пола `h / sin(угол_строки)`; зона «намного дальше ожидаемой» или `status=255` → обрыв/ступенька; «намного ближе» → бугор. ST советует делить матрицу: нижние строки — ямы/ступени, верхние — препятствия над полом. ⚠ Физика: на скользящем угле к полу сигнал слабеет, зеркальный пол может не вернуть эхо — «нет эха» может быть и реальным обрывом, и тёмным/глянцевым полом; дизамбигуация по reflectance/signal.
- **Проекция зоны в точку:** таблицы pitch/yaw на зону (`VL53L5_Zone_Pitch8x8/Yaw8x8`), затем `z=d·cos(pitch)·cos(yaw)`, `x=d·cos(pitch)·sin(yaw)`, `y=d·sin(pitch)`. Готовый пример: `ferrolho/VL53L5CX-BNO08X-viewer` (64 луча + IMU-кватернион + EMA-фильтр + RANSAC-плоскость).
- **Локальная карта/occupancy grid:** кадр → точки → трансформ по позе из одометрии (энкодеры + курс) → растеризация в скользящий 2D-costmap. В ROS2 — `PointCloud2` → `nav2_costmap_2d` (voxel layer). Мост: `adityakamath/tof_imager_ros` (и micro-ROS-версия).
- **Проходы:** найти подряд идущие столбцы с чистой дистанцией > lookahead, принять проход если `2·d·tan(полуугол) > ширина_робота + запас`.
- **Фильтрация:** сначала status-гейт; EMA по зоне между кадрами; отсев одиночных шумных зон по согласию соседей (известный баг — зоны в открытое пространство не сбрасываются в 0); взвешивание по reflectance/signal.
- **vs LiDAR:** дёшево/мало/экономно, плотная карта, но всего 64 луча на ~60°, ~3.5 м, засветка и зеркальные провалы → **хватает для реактивной локальной автономии** (обход, обрыв, локальный costmap с одометрией), но **не замена LiDAR для полного SLAM**.

## 5. Аудит нашего кода (`xiao_tof.h`) — что правильно / что улучшить

**✅ Делаем правильно:**
- Профили с корректными пределами: 4×4 до 60 Гц, 8×8=15 Гц (не нарушаем кап).
- CONTINUOUS для fast/balanced/accurate, AUTONOMOUS+integration только для long — соответствует рекомендациям.
- `status∈{5,9}` как валидность — стандартный фильтр.
- `gTofResults` статический, не на стеке loopTask — правильно (струкура ~1.4 КБ).
- LPn HIGH (перемычка), порядок resolution→mode→freq→start — по UM3038.
- Медианная фильтрация (3/5) — разумно против одиночных выбросов.

**🔧 Стоит улучшить (по убыванию пользы):**
1. **`target_order = CLOSEST`** — мы не задаём, значит дефолт STRONGEST. Для робота ближайшее препятствие в зоне важнее яркого фона. Один вызов `vl53l7cx_set_target_order(CLOSEST)` в init — **самый большой выигрыш для безопасности**.
2. **I2C 1 МГц** вместо 400 кГц — срежет наш ~2–3 с init вдвое (проверить реальную SCL на ESP32-S3; откатиться при сбоях).
3. **Поднять I2C timeout** на ESP32 (clock-stretching ~10–13 мс вешает шину) — гигиена против зависаний.
4. **Фильтр по `range_sigma_mm`/`signal`** — сейчас берём только status; добавить отсев зон с высоким sigma уберёт «дрожащие» ложные близкие значения (особенно важно, т.к. tof_mm = минимум — выброс вниз опасен ложным стопом).
5. **Геометрический отсев пола** вместо грубой «центральной полосы»: считать ожидаемую дистанцию до пола по строке и углу установки — точнее и открывает cliff-detection.
6. **Кросстолк <60 см:** если поставим защитное стекло — сделать Xtalk-калибровку; сейчас (открытый сенсор) просто помнить, что значения <60 см менее надёжны.
7. **На будущее (brain/):** проекция зон в точки + occupancy grid с одометрией — путь к локальной автономии; готовые референсы выше.

**⚠ Проверить против нашей ревизии:** дефолт sharpener (5% vs 14%), нижний предел integration time (2 мс), точные мА потребления (Table 14 DS13865).
