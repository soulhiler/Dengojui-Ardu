# Разводка и пины: как сенсоры сходятся с текущим железом

> Проверка «по пинам»: помещаются ли подобранные в [sensors-2026.md](sensors-2026.md) датчики на
> реальные выводы XIAO ESP32-S3, с учётом того, что уже занято камерой/миком/приводом.
> Главный вывод: **на голом XIAO свободен ровно один пэд (D10)** — весь сенсорный набор требует
> либо I²C-расширителей шины, либо выноса сенсоров на Pi (гибрид). Дата: июнь 2026.

## 1. Карта пинов XIAO ESP32-S3 Sense

Плата даёт **только 11 пользовательских пэдов D0–D10**. Камера/мик/SD подключены через
внутренний B2B-разъём Sense и используют GPIO, которые **на гребёнку не выведены**.

**Внутренние (заняты платой, недоступны):**
| Узел | GPIO |
|---|---|
| Камера OV2640 (DVP) | 10, 11, 12, 13, 14, 15, 16, 17, 18, 38, 39, 40, 47, 48 |
| PDM-микрофон | 41 (DATA), 42 (CLK) |
| Пользовательский LED | 21 |

**Пэды D0–D10 (всё, что доступно для приводов и сенсоров):**
| Пэд | GPIO | Дефолтная функция шины | ADC |
|---|---|---|---|
| D0 | 1 | — | ADC1 |
| D1 | 2 | — | ADC1 |
| D2 | 3 | — | ADC1 |
| D3 | 4 | — | ADC1 |
| D4 | 5 | **I²C SDA** | ADC1 |
| D5 | 6 | **I²C SCL** | ADC1 |
| D6 | 43 | **UART TX** | — |
| D7 | 44 | UART RX / SPI SS | — |
| D8 | 7 | **SPI SCK** | ADC1 |
| D9 | 8 | **SPI MISO** | ADC1 |
| D10 | 9 | SPI MOSI | ADC1 |

## 2. Что уже занято приводом (по `drive_config.h`)

| Пэд | GPIO | Назначение сейчас | Это же — дефолт шины |
|---|---|---|---|
| D0 | 1 | TB6612 **AIN1** | — |
| D1 | 2 | TB6612 **AIN2** | — |
| D2 | 3 | TB6612 **BIN1** | — |
| D3 | 4 | TB6612 **BIN2** | — |
| D4 | 5 | TB6612 **PWMB** (правый) | ⛔ занял **I²C SDA** |
| D5 | 6 | TB6612 **PWMA** (левый) | ⛔ занял **I²C SCL** |
| D6 | 43 | **ENC_L_A** | ⛔ занял UART TX |
| D7 | 44 | **ENC_L_B** | ⛔ занял UART RX/SS |
| D8 | 7 | **ENC_R_A** | ⛔ занял SPI SCK |
| D9 | 8 | **ENC_R_B** | ⛔ занял SPI MISO |
| **D10** | **9** | **— СВОБОДЕН —** | (SPI MOSI) |

### Вывод
- **Занято 10 из 11 пэдов. Свободен только D10 (GPIO9).**
- Привод съел **ровно дефолтные пины I²C (D4/D5), UART (D6/D7) и SPI (D8/D9)**.
- Один свободный пэд ≠ I²C (нужны 2 пина). **Без перепланировки ни один I²C-сенсор не подключить.**
- Аналоговых входов тоже нет в запасе: ADC1-пины (GPIO1–9) все заняты, а ADC2 нельзя при Wi-Fi.

То есть набор «Этап 1» из sensors-2026.md (ToF + IMU + INA по I²C) **физически не влезает** на
голый XIAO поверх текущего привода. Нужен один из двух путей.

## 3. I²C-адреса: где конфликты (если строить общую шину)

| Устройство | Адрес | Конфликт? |
|---|---|---|
| VL53L0X / L1X / L5CX | 0x29 (все!) | ⛔ **да** — несколько ToF на одной шине требуют **XSHUT-секвенирование** (по 1 GPIO на датчик) или мультиплексор **TCA9548A** (0x70, 8 каналов) |
| INA226 / INA3221 | 0x40–0x4F (A0/A1) | нет (адресуемые) |
| BME280 | 0x76/0x77 | нет |
| BH1750 | 0x23/0x5C | нет |
| BNO085 | 0x4A/0x4B | ⚠️ I²C на ESP32-S3 нестабилен → брать по **SPI/UART-RVC**, не I²C |
| PCA9685 (PWM-расш.) | 0x40+ | согласовать с INA |
| MCP23017 (GPIO-расш.) | 0x20–0x27 | нет |

## 4. Путь A — уместить сенсоры на самом XIAO (через расширители)

Идея: **снять с нативных пинов 4 статичных направляющих TB6612** и повесить их на I²C-расширитель,
вернув D4/D5 под I²C-шину. Быстрый ШИМ (20 кГц) оставить на нативных пинах (расширитель PCA9685
не тянет 20 кГц — только ~1.6 кГц, для PWM мотора это гудение, поэтому ШИМ — нативный).

**Предлагаемая новая раскладка:**
| Пэд | GPIO | Новое назначение |
|---|---|---|
| D0 | 1 | TB6612 **PWMA** (нативный LEDC 20 кГц) |
| D1 | 2 | TB6612 **PWMB** (нативный LEDC 20 кГц) |
| D2 | 3 | свободен → бампер / XSHUT / UART-RX LiDAR |
| D3 | 4 | свободен → бампер / GPS-RX |
| **D4** | 5 | **I²C SDA** (общая шина) |
| **D5** | 6 | **I²C SCL** (общая шина) |
| D6 | 43 | ENC_L_A |
| D7 | 44 | ENC_L_B |
| D8 | 7 | ENC_R_A |
| D9 | 8 | ENC_R_B |
| D10 | 9 | свободен → BNO085 (SPI/UART) или ещё один датчик |

**На I²C-шине (D4/D5):** MCP23017 (4 линии AIN1/AIN2/BIN1/BIN2 + запас под бамперы/XSHUT) +
TCA9548A (мультиплексор для ToF-семейства) + INA226 + BME280 + BH1750.
**Что нужно докупить:** MCP23017 (~$1–2), TCA9548A (~$2), сами датчики.
**Минусы:** направляющие моторов идут через I²C (задержка ~мс — для статичного направления ок,
но «горячий» аварийный реверс чуть медленнее); шина перегружена; BNO085 всё равно по SPI/UART
конфликтует с пинами энкодеров (SPI занят) → остаётся только UART-RVC на D2/D3/D10.

**Это работает, но XIAO остаётся «впритык».** Любой LiDAR/GPS/depth — уже не сюда.

## 5. Путь B — гибрид (рекомендуется): сенсоры на Pi, ESP32 на моторике

Совпадает с [планом интеграции Pi Zero](pi-zero-integration.md) и снимает всю тесноту по пинам:
**привод и энкодеры остаются на XIAO как есть (ничего не переразводим), а сенсоры садятся на Pi.**

У Raspberry Pi 40-pin: аппаратные I²C (GPIO2/3), SPI, **2+ UART**, много свободных GPIO — там
весь набор размещается свободно:
| Сенсор | Шина на Pi |
|---|---|
| ToF (VL53L5CX/L1X), INA226, BME280, BH1750 | I²C (GPIO2/3) + при нужде TCA9548A |
| BNO085 | SPI или UART-RVC (на Pi I²C тоже работает корректно) |
| 2D-LiDAR (RPLIDAR C1 / LD19) | UART/USB |
| GPS NEO-M8N | UART |
| depth-камера (OAK-D/RealSense) | USB |
| бампер, cliff-XSHUT | свободные GPIO |

ESP32 ↔ Pi — по Wi-Fi (этап 1) или USB-serial (этап 2), как в плане. ESP32 при этом может отдать
Pi энкодеры/телеметрию, а Pi — слать `/drive`. **Пины XIAO не трогаем вообще.**

## 6. Питание и уровни (общее для обоих путей)

- **5 В vs 3.3 В:** XIAO I²C/GPIO — 3.3 В. HC-SR04 ECHO (5 В) → делитель (уже отмечено в
  `drive_config.h.example`). US-100/ToF/IMU/INA/BME — 3.3 В нативно, делитель не нужен.
- **TB6612:** логика VCC 2.7–5.5 В, VM — отдельное питание моторов, общая GND.
- **Токи:** I²C-датчики — единицы мА; LiDAR (моторчик) — сотни мА по 5 В (нужен 5 В-рейл и запас
  по БП); depth-камеры по USB — это уже бюджет Pi (см. [research-boards-2026.md](research-boards-2026.md)).
- **Общая земля** обязательна между XIAO, драйвером, Pi и сенсорами.

## 7. Итоговая рекомендация по этапам

1. **Сейчас (минимум изменений):** включить уже разведённые **бампер** (нужен лишь GPIO — свободен
   только D10; либо освободить пин) и при желании ультразвук на D10/делитель. Полноценный I²C-набор
   на голый XIAO не вешать — пинов нет.
2. **Если хочется сенсоры именно на ESP32** → Путь A: добавить MCP23017 + TCA9548A, перевести
   направляющие TB6612 на расширитель, вернуть I²C на D4/D5. Перепланировка `drive_config.h`.
3. **Рекомендуемый путь** → Путь B (гибрид): сенсоры на Pi, XIAO не переразводим. Это снимает
   дефицит пинов, даёт место под LiDAR/GPS/depth и совпадает с планом автономности.

> Практический смысл: дефицит пинов на XIAO — ещё один аргумент за гибридную архитектуру MCU+SBC.
> Навешивать весь сенсорный набор на один XIAO технически можно только через I²C-расширители и
> ценой тесноты; разнести моторику (ESP32) и восприятие (Pi) — чище и масштабируемее.
