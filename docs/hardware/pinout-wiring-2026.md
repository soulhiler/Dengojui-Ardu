# Разводка и пины: как сенсоры сходятся с текущим железом

> Проверка «по пинам»: помещаются ли подобранные в [sensors-2026.md](sensors-2026.md) датчики на
> реальные выводы XIAO ESP32-S3, с учётом того, что уже занято камерой/миком/приводом.
> Дата: июнь 2026.
>
> **⚠️ Актуальная распайка платы — [`wiring-xiao-motor-tof.md`](wiring-xiao-motor-tof.md)** (это
> канон, синхронизирован с `drive_config.h`). Этот документ — анализ дефицита пинов и вариантов
> расширения; ниже карта пинов приведена в соответствие с реально зашитой раскладкой (I²C на
> D9/D10, ToF + IMU на общей шине, энкодеры не установлены).

## 1. Карта пинов XIAO ESP32-S3 Sense

Плата даёт **только 11 пользовательских пэдов D0–D10**. Камера/мик/SD подключены через
внутренний B2B-разъём Sense и используют GPIO, которые **на гребёнку не выведены**.

**Внутренние (заняты платой, недоступны):**
| Узел | GPIO |
|---|---|
| Камера OV2640 (DVP) | 10, 11, 12, 13, 14, 15, 16, 17, 18, 38, 39, 40, 47, 48 |
| PDM-микрофон | 41 (DATA), 42 (CLK) |
| Пользовательский LED | 21 |

**Пэды D0–D10 (всё, что доступно для приводов и сенсоров):**
| Пэд | GPIO | «Дефолтная» функция шины | ADC |
|---|---|---|---|
| D0 | 1 | — | ADC1 |
| D1 | 2 | — | ADC1 |
| D2 | 3 | — | ADC1 |
| D3 | 4 | — | ADC1 |
| D4 | 5 | I²C SDA | ADC1 |
| D5 | 6 | I²C SCL | ADC1 |
| D6 | 43 | UART TX | — |
| D7 | 44 | UART RX / SPI SS | — |
| D8 | 7 | SPI SCK | ADC1 |
| D9 | 8 | SPI MISO | ADC1 |
| D10 | 9 | SPI MOSI | ADC1 |

> **Важно про «дефолтные функции».** На ESP32-S3 I²C/SPI/UART **не привязаны жёстко** к этим
> пинам — GPIO-матрица заводит периферию на любой пэд (`Wire.begin(sda, scl)` с произвольными
> номерами). Жёстко зафиксированы только ADC-каналы, USB D±/(GPIO19/20) и strapping-пины. Поэтому
> столбец выше — лишь «значение по умолчанию» из доков Seeed, а не ограничение: проект как раз
> увёл I²C с D4/D5 на **D9/D10**, освободив D4/D5 под ШИМ моторов.

## 2. Что занято сейчас (по `drive_config.h`, прошивка 1.4.x)

| Пэд | GPIO | Назначение сейчас |
|---|---|---|
| D0 | 1 | TB6612 **AIN1** (левый, направление) |
| D1 | 2 | TB6612 **AIN2** (левый, направление) |
| D2 | 3 | TB6612 **BIN1** (правый, направление) |
| D3 | 4 | TB6612 **BIN2** (правый, направление) |
| D4 | 5 | TB6612 **PWMB** (правый, ШИМ 20 кГц) |
| D5 | 6 | TB6612 **PWMA** (левый, ШИМ 20 кГц) |
| D6 | 43 | **— свободен —** (энкодеры не установлены) |
| D7 | 44 | **— свободен —** (энкодеры не установлены) |
| D8 | 7 | **ToF LPn** (управляемый сброс/recovery шины I²C) |
| D9 | 8 | **I²C SDA** — общая шина: ToF + MPU6050 |
| D10 | 9 | **I²C SCL** — общая шина: ToF + MPU6050 |

### Вывод (как есть на железе)
- **Привод (6 пинов) — D0–D5.** I²C-шина — **D9/D10**, LPn ToF — **D8**.
- На **одной шине D9/D10** живут сразу **два устройства**: VL53L7CX (0x29) + MPU6050 (0x68) —
  адреса не конфликтуют, проверено на железе (400 кГц). Лишних пинов под IMU **не понадобилось**.
- **Энкодеры в текущей сборке не установлены** (`DRIVE_ENC_* = 0`) → пэды **D6, D7 фактически
  свободны**. Это запас под будущий датчик/сигнал.
- То есть набор «Этап 1» из sensors-2026.md (ToF + IMU по I²C) **уместился на голый XIAO без
  расширителей** — за счёт того, что (а) I²C переназначен на D9/D10, (б) энкодеры пока не ставятся.

### Когда дефицит пинов вернётся
Тесно станет, если **одновременно** захотеть: энкодеры AB на оба колеса (4 пина), бампер,
ультразвук, плюс что-то по UART/SPI. На 11 пэдах поверх привода это уже не помещается без
переразводки — тогда в игру входят Путь A (I²C-расширители) или Путь B (гибрид с Pi) ниже.
Частный случай уже встречался: правый энкодер B-канал конфликтовал с SDA (GPIO8) и был отключён
(см. `wiring-xiao-motor-tof.md`).

## 3. I²C-адреса: что на шине сейчас и что добавится

| Устройство | Адрес | Статус |
|---|---|---|
| **VL53L7CX** (ToF) | 0x29 | ✅ на шине сейчас |
| **MPU6050** (IMU курса) | 0x68 (0x69 при AD0=1) | ✅ на шине сейчас, не конфликтует с ToF |
| BNO085 (апгрейд курса) | 0x4A/0x4B | сядет на ту же шину без конфликта (см. `orientation-and-mapping-decisions.md`) |
| INA226 / INA3221 | 0x40–0x4F (A0/A1) | нет конфликта (адресуемые) |
| BME280 | 0x76/0x77 | нет |
| BH1750 | 0x23/0x5C | нет |
| Несколько ToF (VL53L*) | 0x29 у всех! | ⛔ требуют **XSHUT/LPn-секвенирования** (по 1 GPIO на датчик) или мультиплексора **TCA9548A** (0x70) |

> Прежняя версия документа советовала BNO085 «только по SPI/UART, т.к. I²C на ESP32-S3 нестабилен».
> На практике проект гоняет ToF + MPU6050 по I²C на 400 кГц стабильно; BNO085 штатно работает по
> I²C на 0x4A. Так что I²C — рабочий путь и для него (см. решение в `orientation-and-mapping-decisions.md`).

## 4. Путь A — навесить ещё сенсоров на сам XIAO (через расширители)

Если в будущем поверх привода понадобится большой I²C-набор + энкодеры одновременно — снять
часть статичных линий с нативных пинов и вынести их на расширитель, а быстрый ШИМ (20 кГц)
оставить на нативных (PCA9685 тянет лишь ~1.6 кГц — для ШИМ мотора это гудение).

**На I²C-шине (D9/D10):** к ToF+IMU добавить TCA9548A (мультиплексор для нескольких ToF) +
MCP23017 (GPIO-расширитель под направляющие/бамперы/XSHUT) + INA226 + BME280 + BH1750.
**Докупить:** MCP23017 (~$1–2), TCA9548A (~$2), сами датчики.
**Минусы:** статичные линии через I²C добавляют ~мс задержки; шина перегружается; «горячий»
аварийный реверс чуть медленнее. Работает, но XIAO остаётся «впритык» — LiDAR/GPS/depth уже не сюда.

## 5. Путь B — гибрид (рекомендуется на перспективу): сенсоры на Pi, ESP32 на моторике

Совпадает с разбором кандидатов SBC ([candidate-1-budget-compute.md](candidate-1-budget-compute.md))
и снимает всю тесноту по пинам: **привод/I²C на XIAO остаются как есть, тяжёлые сенсоры садятся на Pi.**

У Raspberry Pi 40-pin: аппаратные I²C (GPIO2/3), SPI, 2+ UART, много свободных GPIO — весь набор
размещается свободно:
| Сенсор | Шина на Pi |
|---|---|
| ToF (VL53L7CX/L5CX), INA226, BME280, BH1750 | I²C (GPIO2/3) + при нужде TCA9548A |
| BNO085 | I²C / SPI / UART-RVC |
| 2D-LiDAR (RPLIDAR C1 / LD19) | UART/USB |
| GPS NEO-M8N | UART |
| depth-камера (OAK-D/RealSense) | USB |
| бампер, cliff-XSHUT | свободные GPIO |

ESP32 ↔ Pi — по Wi-Fi (этап 1) или USB-serial (этап 2). Конкретная разводка гибрида —
[wiring-path-b.md](wiring-path-b.md). **Пины XIAO при этом не трогаем.**

## 6. Питание и уровни (общее для обоих путей)

- **5 В vs 3.3 В:** XIAO I²C/GPIO — 3.3 В. HC-SR04 ECHO (5 В) → делитель (отмечено в
  `drive_config.h.example`). ToF/IMU/INA/BME — 3.3 В нативно, делитель не нужен.
  ⚠️ VIN/LPn ToF — **только 3.3 В** (на 5 В подтяжки уйдут на 5 В, пины ESP32-S3 не 5V-tolerant).
- **TB6612:** логика VCC 2.7–5.5 В, VM — отдельное питание моторов, общая GND.
- **Токи:** I²C-датчики — единицы мА; LiDAR (моторчик) — сотни мА по 5 В (нужен 5 В-рейл и запас
  по БП); depth-камеры по USB — это уже бюджет Pi (см. [research-boards-2026.md](research-boards-2026.md)).
- **Общая земля** обязательна между XIAO, драйвером, Pi и сенсорами.

## 7. Итоговая рекомендация по этапам

1. **Сейчас (как на железе):** ToF + IMU уже на общей шине D9/D10, привод на D0–D5, LPn на D8.
   D6/D7 свободны (энкодеры не ставятся) — туда можно повесить бампер/один цифровой сигнал.
2. **Если расширять сенсорику на самом ESP32** → Путь A: TCA9548A (для нескольких ToF) и/или
   MCP23017, INA/BME/BH по той же шине. Энкодеры на оба колеса + большой набор разом — уже впритык.
3. **На перспективу (зрение/LiDAR/SLAM)** → Путь B (гибрид): тяжёлые сенсоры на Pi, XIAO не
   переразводим. Снимает дефицит пинов и совпадает с планом автономности.

> Практический смысл: на текущем минимуме (ToF + IMU) пинов **хватает** — спасли GPIO-матрица
> (I²C на D9/D10) и отказ от энкодеров. Но это «впритык»: любой следующий крупный слой восприятия
> (LiDAR/GPS/depth или полные энкодеры + бамперы) — аргумент за гибридную архитектуру MCU + SBC.
