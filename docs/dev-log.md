## 2026-05-18 — mDNS-автопоиск платы в приложении (IP больше не нужен)

По запросу: приложение само находит плату, IP вводить не надо.

- **Прошивка без изменений** — уже рекламирует `MDNS.begin("xiao-cam")` +
  `addService("http","tcp",80)` → сервис `_http._tcp` инстанс `xiao-cam`.
- **`BoardDiscovery.kt`** (новый): Android NSD — `discoverServices
  ("_http._tcp.")`, фильтр имени по «xiao», `resolveService` → текущий
  IPv4 платы. MulticastLock на время поиска; single-resolve guard
  (ограничение NsdManager до API 34); таймаут; всё в main-поток, один раз.
- **`MainActivity`**: поле пустое / `xiao-cam.local` / `auto` → mDNS-поиск
  (6 с); найдено → старт каналов на найденном IP; не найдено → пробуем
  `last_ip`, иначе просим ввести IP вручную. Литеральный IP → прямой путь
  (ручной фолбэк). `disconnect`/`onDestroy` останавливают поиск.
  Дефолт поля → `xiao-cam.local`; устаревший `.17` мигрирует на авто.
- **Manifest**: +`ACCESS_WIFI_STATE`, +`CHANGE_WIFI_MULTICAST_STATE`.
- Файлы: `BoardDiscovery.kt` (new), `MainActivity.kt`,
  `AndroidManifest.xml`, `strings.xml`, `activity_main.xml`.

**Сборка:** PowerShell-инструмент в сессии завис; APK собран
`java -cp gradle/wrapper/gradle-wrapper.jar
org.gradle.wrapper.GradleWrapperMain assembleDebug` → BUILD SUCCESSFUL,
Kotlin/NSD скомпилировались чисто (7.2 МБ). Поведение NSD на самом
телефоне — подтвердить на устройстве.

**Для пользователя:** поставить новый APK → просто «Подключить»
(IP не вводить; поле можно оставить `xiao-cam.local`).

## 2026-05-18 — Приложение слало на .17; дефолтный IP исправлен

**Симптом:** на экране статус `видео: Failed to connect to /192.168.9.17:80`;
поле IP не редактировалось (работали только кнопки). Браузер телефона по
`http://192.168.9.18/capture` — фото ОК (сеть и камера исправны).

**Причина:** дефолтный IP в приложении был захардкожен `192.168.9.17`
(и мог «прилипнуть» в SharedPreferences после прошлого Connect), а плата
по DHCP получила `192.168.9.18`. Приложение стучалось не туда. Трудность
с правкой поля усугубляла.

**Фикс (приложение):** дефолт хоста → `192.168.9.18`; авто-миграция
сохранённого устаревшего `.17` → `.18` (`MainActivity.onCreate`);
`activity_main.xml` дефолт-текст → `.18`. APK пересобран.

**Сборочное:** инструмент PowerShell в сессии завис (любые команды exit 1
без вывода), `gradlew.bat` через cmd не отрабатывал. APK собран
shell-агностично: `java -cp gradle/wrapper/gradle-wrapper.jar
org.gradle.wrapper.GradleWrapperMain assembleDebug` → BUILD SUCCESSFUL.

**Durable-рекомендация:** DHCP-резервация на роутере
(MAC `E8:F6:0A:8B:C3:84` → `192.168.9.18`) — тогда IP постоянный и поле
вообще не трогать. Опционально на будущее: Android NSD/mDNS
(`xiao-cam.local`), чтобы IP стал неважен.

**Установка:** удалить старое приложение → поставить новый APK → открыть →
«Подключить» (вводить ничего не нужно).

## 2026-05-18 — «Камера не подключается»: OV3660 QXGA→FB-OVF, исправлено

**Симптом:** плата в WiFi, но в приложении нет видео; `/capture` → HTTP 500.

**Диагноз на железе:** сенсор OV3660 (`cam_pid=0x3660`) был сконфигурирован на
**QXGA + jpeg_quality=2** (почти максимум). На непрерывном MJPEG это
переполняет буфер кадров → бесконечный `cam_hal: FB-OVF` →
`esp_camera_fb_get()` стабильно `NULL`. Телеметрия: `cam_fail`растёт,
`cam_frames_stream=0`, `/capture` отдаёт 500 «capture failed» (14 Б).
Т.е. ранняя пометка «FB-OVF безвреден» оказалась неверной под нагрузкой —
это и есть причина отсутствия видео (Фаза 2, всплыло только на железе).

**Фикс** (`xiao_cam_stream/initCamera`): стабильный потоковый профиль —
`config`: `FRAMESIZE_SVGA` + `jpeg_quality=12` (fb_count=2, GRAB_LATEST);
пост-init для OV3660/OV2640: `set_framesize(SVGA)` + `set_quality(12)`
(было QXGA/UXGA @ q2). Разрешение можно поднять позже, если поток стабилен.

**Проверка на железе (после reflash):** `FB-OVF=0`, `cam_fail=0`,
`/capture` → 200, **JPEG ~21 КБ** (3/3); `/stream` → ~1 МБ/4 с,
`cam_frames_stream=48` → **≈12 fps**. Плата `192.168.9.18`,
`Duangdeehouse2  2.4GHz`, RSSI −46. Камера и поток стабильны.

**Для приложения:** IP = `192.168.9.18`, телефон в той же 2.4 ГГц сети.

## 2026-05-18 — Видео/аудио в приложении не стартовали → плата выведена в WiFi

**Симптом:** на смартфоне приложение открылось, но видео и звук не идут.

**Диагноз (по serial, не догадки):** приложение ни при чём — **плата не в
WiFi**. Сначала была прошита с плейсхолдер-`secrets.h` (`YOUR_WIFI_PASSWORD`),
затем — с неверным написанием SSID. `wifi_status=disconnected`, `ip=0.0.0.0`.

**Решение:** временным скетчем-сканером считал эфир радио платы →
**точный SSID = `Duangdeehouse2  2.4GHz`** (ДВА пробела перед «2.4GHz»;
RSSI −41, WPA2). Прописал точный SSID + пароль в `xiao_cam_stream/secrets.h`
(gitignored, в репо не идёт; pre-commit хук тоже блокирует), перепрошил.

**Результат:** `wifi_status=connected`, **IP `192.168.9.18`**, RSSI −43.
Сквозная проверка по WiFi с ПК: `GET /telemetry` → 200,
`GET /capture` → 200, **JPEG 370 КБ**. `cam_fail=0` — простойный
`cam_hal: FB-OVF` подтверждённо безвреден (полный кадр отдаётся).

**Для приложения:** ввести IP **`192.168.9.18`** (дефолт в приложении
`192.168.9.17` — мимо на единицу), телефон в сети `Duangdeehouse2  2.4GHz`.
Альтернатива на будущее: mDNS `xiao-cam.local` или DHCP-резервация
(чтобы IP не «плавал»). Креды в git не попали (только этот dev-log).
